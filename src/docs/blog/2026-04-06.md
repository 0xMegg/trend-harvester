# 2026-04-06 — Epic v5.1, 멀티리포 오케스트레이션 완성

> 단일 리포 전용이던 Epic 시스템을 멀티리포까지 확장 — 커밋 오케스트레이션과 모니터링까지 갖췄다.

## 배경

Claude Code Harness Template의 Epic/Task 자동화 스크립트는 v5에서 병렬 Slice 실행을 도입했지만, 단일 Git 리포지토리 기준으로 설계되어 있었다. haink처럼 backend/frontend가 별도 리포인 멀티리포 프로젝트에서는 커밋이 한쪽 리포에서만 실행되거나, 로그가 섞이는 문제가 있었다.

## 작업 내역

### Epic v5.1 — 멀티리포 지원 + 커밋 오케스트레이션 + 모니터링

가장 큰 변경. 13개 파일에 걸친 전면 업그레이드:

- **run-epic.sh / run-task.sh**: 멀티리포 환경 감지, 각 하위 리포에서 독립적으로 커밋/푸시 실행
- **epic.md / task.md 커맨드**: 멀티리포 커밋 규칙과 scope 명시
- **role-planner.md**: 멀티리포 Slice 설계 시 리포 경계 명시 가이드
- **role-reviewer.md**: 리포별 독립 리뷰 + 커밋 오케스트레이션 프로토콜
- **verify.md / epic-plan.md / handoff.md**: 멀티리포 검증 및 핸드오프 양식
- **working-rules.md / git.md**: 멀티리포 Git 규칙 추가

### fix: Epic 소수점 버전 인식

Epic 번호에 소수점이 포함된 경우(예: `3.1`) regex가 인식하지 못하는 버그 수정. `run-epic.sh`의 패턴 매칭을 `[0-9]+(\.[0-9]+)?`로 확장했다.

### fix: Epic별 고유 로그 디렉토리

이전 Epic 실행의 로그가 다음 Epic 실행 시 오염되는 문제 수정. Epic마다 고유한 로그 디렉토리를 생성하도록 `run-epic.sh`와 `run-task.sh`를 수정했다.

v5.1은 haink 프로젝트에서 실제로 멀티리포 Epic을 돌리면서 발견한 문제들을 즉시 템플릿에 역반영한 결과물이다. 특히 커밋 오케스트레이션(각 리포에서 독립적으로 lint → test → commit → push)은 실전에서 바로 효과를 봤다 — haink Epic 3 검증이 깔끔하게 통과한 게 그 증거다.
