# Harness Forge — Downstream 피드백 리포트

**From**: kody-workspace (하네스 다운스트림)
**Date**: 2026-04-23
**Trigger**: Epic 7 (3시간 20분, 14개 Slice, parallel Stage 4개) 실행 중 관찰된 하네스 레이어 결함
**목적**: 범용 하네스 템플릿(포지) 수준에서 고치면 다른 다운스트림에서도 재발 방지되는 8개 항목 공유

---

## TL;DR

Epic 실행이 **형식상 성공**(14/14 Slice APPROVE)으로 종료됐으나, 실제로는 Epic acceptance criteria 미달(Blocker 2건 잔존). 실행 도중 monitor 사망·git push silent fail·UI verification 프로세스 누수도 동반. 근본 원인의 대부분이 프로젝트 특수성이 아닌 **하네스 템플릿 레이어의 구조적 결함**이며, 같은 패턴이 다른 다운스트림에서 반복될 가능성 큼.

**요청**: P0 4건(template 3 + scripts 1)을 다음 포지 마이너 버전에 포함.

## 1. 관찰된 사건 (범용 서술)

### E1. Planner가 primitive 확장 slice와 call-site 업데이트 slice를 분리
- 한 slice가 공용 컴포넌트에 새로운 prop/flag를 추가. 후속 slice들은 **그 prop을 세팅하는 호출자 파일을 Files 목록에 포함하지 않음**.
- 결과: prop은 support 됐으나 실제 세팅 0건 → 기능이 사실상 dead code. acceptance-time에 발견.
- **재현 조건**: "layer order(data → logic → UI)" 원칙이 "새 API는 call-site 업데이트 없이 완성된다"로 잘못 해석될 때.

### E2. 토큰 체계 도입 후 기존 하드코딩의 전수 파악이 plan 단계에 없음
- 새 토큰 시스템을 도입한 slice 이후, 페이지 파일에는 여전히 토큰을 우회하는 인라인 literal(스타일 값 등)이 5곳 남아 있었음.
- `grep` 한 번이면 발견 가능했으나 Planner Pre-Start Checklist에 해당 단계가 없어 plan에 누락.
- **재현 조건**: 마이그레이션/리팩토링 성격의 Epic 전반.

### E3. Slice acceptance ≠ Epic acceptance — 최종 audit slice가 ITERATE 평결을 내도 Epic 파이프라인이 성공으로 종결
- 마지막 verification/audit slice가 정직하게 "Blocker=2, Verdict: ITERATE"를 리포트로 남겼음.
- 그러나 slice 자체는 "리포트를 잘 썼다"는 이유로 APPROVE. Epic wrapper는 모든 slice APPROVE를 성공으로 처리 → Epic 완료 알림.
- 사용자가 audit 본문을 열어보지 않았다면 실패를 인지 못 할 수 있음.
- **재현 조건**: audit/QA slice를 마지막 Stage에 두는 모든 Epic.

### E4. `/epic` monitor 명령이 zsh에서 parse error로 즉시 사망
- `for d in "$PATH"/task-slice-*/ 2>/dev/null; do` — zsh는 for 헤더의 리다이렉션을 허용하지 않음. bash는 허용.
- 3시간 20분 run 동안 실시간 진행 가시성 0. monitor의 존재 이유가 사라짐.
- **재현 조건**: zsh 기본 셸 환경(현재 macOS 기본).

### E5. origin remote가 없어도 Epic이 정상 종결
- `git remote -v` 비어 있는 환경에서 실행. finalize 단계의 `git push`가 `|| echo "push skipped or failed"`로 silent 처리.
- 결과: 20여 개 커밋이 로컬에만 존재. 사용자가 Epic 종료 후에야 인지.
- **재현 조건**: 새 저장소·로컬 개발·migrating 단계 프로젝트.

### E6. Verification 용도의 장기 프로세스(dev server 등) 누수
- Developer 또는 Reviewer가 UI 검증을 위해 띄운 장기 프로세스(Next dev server 등)가 slice 종료 후 kill되지 않음.
- 최소 1시간 이상 CPU/포트 점유. 외부에서 `pkill` 하면 역할 종료되지만 자동 정리 경로 없음.
- **재현 조건**: UI/frontend verification이 필요한 모든 slice.

### E7. Reviewer가 "새로 추가된 public API가 한 번도 사용되지 않는" 상태를 APPROVE로 승인
- E1과 짝. Reviewer가 slice 범위 내 diff만 보고 판정했고, 새 prop이 call-site 0건인 상태를 체크하지 않음.
- Reviewer template에 해당 가드가 없음.

### E8. Slice review time이 중앙값의 5~7배까지 튀는 경우의 진단 불가
- 한 slice의 review가 72분 소요(타 slice 평균 10~15분). 원인은 slice scope(토큰 파급 범위가 3-way theme × 여러 cascade)가 컸기 때문으로 추정되나, review 로그는 1줄짜리 최종 메시지만 남아 있어 시간대별 분석 불가.
- **결과**: "행 상태인가 정상 작업인가" 판단이 로그만으로는 불가, 외부 `ps`로 CPU time 확인해야 함.

### E9. `scripts/run-task.sh` 색상 변수 참조가 선언보다 앞에 있어 `set -u`에서 즉시 exit 1
- `check_harness_version()` 함수 본문이 `${CYAN}`, `${YELLOW}`, `${GREEN}` 등 색상 변수를 참조. 그런데 이 함수가 파일 상단에서 정의·호출되는 반면, 색상 변수 정의 블록은 그 아래에 위치함.
- `set -euo pipefail` 때문에 unbound variable 참조는 즉시 exit 1. 다운스트림이 첫 `/task` 실행 시 `CYAN: unbound variable` 에러로 Developer 역할 진입 자체 실패.
- **확정 재현**: bash + `set -u` 조합 모든 다운스트림. zsh/bash 셸 차이 무관, strict mode 여부만 관련.
- **kody 로컬 우회**: 색상 변수 정의 블록을 `check_harness_version()` 호출 이전으로 hoist하는 patch를 `scripts/run-task.sh`에 적용 (2026-04-23 커밋). 포지에 머지 필요.
- **참고**: `scripts/run-epic.sh`도 동일 구조 — 우연히 color var 정의가 앞에 와서 지금은 무사하지만 같은 결함 재발 가능. 두 스크립트 모두 정의 순서 점검 권장.

### E10. Post-task 세션 작업이 `handoff/latest.md` 갱신을 유도하지 않음 — 설계 gap
- handoff 갱신 책임이 **`/task` flow 내부 Reviewer 역할에만** 묶여 있음. Reviewer APPROVE 시점의 상태가 handoff에 박제됨.
- 같은 세션에서 `/task` 밖의 작업(`gh pr merge`, 브랜치 정리, 다른 리포에 상류 피드백 전달, 후속 환경 정비 등)을 수행해도 **어느 agent도 handoff를 건드리지 않음**.
- 결과: 마지막 APPROVE 시점 스냅샷이 실제 세션 종료 상태와 어긋남. 다음 세션이 handoff를 진입점으로 읽으면 **이미 끝난 작업을 재시도**하거나 상태 판단 오류 발생.
- **kody 재현 (2026-04-23)**: Slice 7.15 APPROVE 후 같은 세션에서 PR #1/#2 머지 + 포지 E9 피드백 전달 완료. handoff는 "APPROVE 직후, dev 머지 대기" 상태로 멈춰 있어 세션 종결 점검 단계에서 stale 확인.
- **확정 재현 조건**: APPROVE 후 같은 세션 안에서 PR merge/브랜치 정리/상류 전달 등 tail 작업을 수행하는 모든 다운스트림 흐름.
- **범주**: 버그 아닌 **설계 gap** — 설계 수준에서 "handoff는 task phase artifact"와 "다음 세션 진입점" 두 역할이 충돌.

## 3. 제안 개선사항

### P0 — Blocker-tier (다음 마이너 버전 포함 요청)

**P0-1. Planner template: "Pre-write grep" 절 (E1, E2)**
- `templates/role-planner.md` 또는 `epic-plan.md`의 Pre-Start Checklist에 아래 단계 명시:
  > Plan 작성 전 다음 grep 결과를 Files 목록·Stage 배치에 반영한다.
  > - 새 prop/flag 도입 시: `grep -rl "<name>" <src_roots>/` 로 call-site 전수 확보
  > - 기존 literal을 토큰으로 대체하는 리팩토링 시: 대체 대상 패턴을 grep (예: `textTransform`, `box-shadow`, 하드코딩 색상)
  > - 결과 파일들이 동일 Stage에 같이 들어가거나 순차 Stage로 분리돼야 함 (overlap gate와 호환)
- **근거**: plan 단계의 coverage 실패가 E1·E2의 공통 원인. 한 줄짜리 규칙이지만 파급 효과가 가장 큼.

**P0-2. Epic plan template: audit slice Done-when을 Epic gate와 기계적으로 연결 (E3)**
- `templates/epic-plan.md`의 "Stages & Slices" 예시 또는 "Epic Acceptance Criteria"에 다음 convention 명시:
  > Epic의 마지막 verification/audit slice는 Done-when에 audit 리포트의 기계 판정(`Verdict: PASS` 또는 `Blocker=0`)을 반드시 포함한다. 이 조건을 만족하지 못하면 slice 판정도 REVISE, Epic 전체 fail 처리.
- **함께 필요**: `scripts/run-epic.sh` 종료 경로에서 audit 리포트를 파싱해 기계적으로 gate를 확인하는 훅(아래 P2-1) — 또는 Reviewer가 ITERATE를 감지하면 자동으로 slice REVISE로 판정하게 하는 규칙.

**P0-3. `/epic` monitor 명령 zsh 호환 패치 (E4)**
- `.claude/commands/epic.md`의 monitor 한 줄 명령
- **현재 (zsh parse error)**:
  ```bash
  for d in "$L"/task-slice-*/ 2>/dev/null; do ...
  ```
- **제안 수정**:
  ```bash
  for d in "$L"/task-slice-*/; do [ -d "$d" ] || continue; ...
  ```
  또는 전체를 `bash -c '...'`로 래핑해 셸 격리.
- **근거**: macOS/zsh 환경의 다운스트림에서 확정 재현.

**P0-4. `run-epic.sh`: origin remote preflight (E5)**
- `setup_epic_branch()` 시작부에 추가:
  ```bash
  if ! git remote | grep -q .; then
    echo "WARN: no git remote configured — commits will stay local only" >&2
    [ "${HARVEST_ALLOW_NO_REMOTE:-0}" = "1" ] || {
      echo "  set HARVEST_ALLOW_NO_REMOTE=1 to proceed, or 'git remote add origin <URL>'" >&2
      exit 1
    }
  fi
  ```
- **근거**: 현재 `git push`가 silent fail하는 설계가 "로컬 전용 프로젝트 배려"일 수 있으나, **확인 없이 silent**인 것이 문제. 환경 변수 escape hatch를 두면 양쪽 요구 모두 충족.

**P0-5. `scripts/run-task.sh` + `run-epic.sh` 색상 변수 선언 순서 버그 (E9)**
- 증상: `check_harness_version()` 실행 중 `${CYAN}: unbound variable` → exit 1. `/task` 진입 시 Developer 역할 기동 실패.
- 원인: 색상 변수 정의 블록이 해당 함수 정의·호출보다 아래에 위치. `set -euo pipefail` 환경에서 확정 재현.
- **제안 수정** (택1):
  1. 포지 `src/scripts/run-task.sh`, `src/scripts/run-epic.sh` 양쪽에서 color var 정의 블록을 `set -euo pipefail` 바로 다음 위치로 hoist (구조적 해결)
  2. 또는 color vars를 함수 내 `: "${CYAN:=}"` 형태로 defensive 초기화 (minimum-change 해결)
- **근거**: 다운스트림이 harness 첫 사용부터 걸림. kody는 로컬 patch로 우회 중이나 다른 다운스트림은 같은 증상으로 막힐 것. P0 재현성.
- **회귀 방지**: 포지 CI에 `bash -n + bash -euo pipefail scripts/run-task.sh --dry-run "smoke"` 추가 권장. shellcheck 통합 시 SC2154(referenced but not assigned) 감지.

### P1 — Major (다음 다음 버전)

**P1-1. Developer template: follow-up call-site 섹션 필수화 (E1 보조)**
- `templates/role-developer.md`에 추가:
  > 새 public prop/export를 추가한 경우, 해당 이름을 `grep -rn "<name>" <src_roots>/` 돌려 **미사용 파일 목록**을 handoff의 별도 섹션(예: `## Follow-up call-sites`)에 기록한다. 다음 Planner가 후속 작업 Files 목록에 이 항목을 반영한다.
- **근거**: P0-1이 plan 단계에서 예방, 이 항목은 실행 단에서 이중 안전망.

**P1-2. Reviewer template: dead-code 가드 (E7)**
- `templates/role-reviewer.md`에 추가:
  > 새로 추가된 public prop/export는 최소 1개 call-site에서 사용 증거가 있어야 APPROVE. 예외: Planner가 Epic plan에 "Stage X에서 사용 예정"을 명시한 경우. 증거 없으면 REVISE.
- **근거**: 다운스트림이 primitive-first refactor를 여러 stage에 걸쳐 하는 경우를 위한 가드.

**P1-3. Developer/Reviewer template: 장기 실행 프로세스 위생 (E6)**
- 양쪽 role template에 추가:
  > verification 용도의 장기 프로세스(dev server, watcher, tunnel 등)는 `run_in_background: true`로만 실행. slice 종료 전 명시적으로 PID를 kill하고, `ps`로 종료 확인. task-status 갱신 전 process leak 금지.
- **근거**: UI/frontend verification 수요가 있는 모든 다운스트림에서 재발.

**P1-4. Reviewer template + 세션 종결 훅: post-task handoff 갱신 유도 (E10)**
- 증상: APPROVE 후 같은 세션에서 수행하는 PR merge / 브랜치 정리 / 상류 피드백 전달 등이 `handoff/latest.md`에 반영되지 않음 → 다음 세션 진입 시 stale.
- **제안 수정** (조합 권장):
  1. **Reviewer template 보강** (`templates/role-reviewer.md`):
     > APPROVE 후 같은 세션에서 추가로 수행한 post-task 작업(PR merge, 브랜치 정리, 다른 리포 전달, 후속 환경 정비 등)은 `handoff/latest.md`의 별도 섹션(예: `## Post-task activities`)에 append 한다. 없으면 해당 섹션을 신규 생성.
  2. **SessionEnd 훅 또는 `scripts/run-task.sh` finalize 경로 보강**: APPROVE 이후 신규 커밋/머지 흔적이 있고 `handoff/latest.md` 수정 시각이 그보다 앞서면 경고 출력 (세션 종료 전에 "handoff 갱신이 필요합니다" 알림).
- **근거**: "task phase artifact인 동시에 다음 세션 진입점"이라는 handoff의 이중 역할을 설계 수준에서 정리. 버그 아닌 설계 gap이라 포지 내 해결이 구조적으로 타당.
- **회귀 방지**: 훅 기반 검증이면 다운스트림이 skip 불가. template 문구만이면 reviewer 판단에 의존.

### P2 — Minor / Nice-to-have

**P2-1. `scripts/acceptance-check.sh` 신규 (E3 자동화)**
- P0-2를 기계적으로 보조하는 스크립트. `outputs/reviews/epic-*-audit.md`에서 `Verdict:` 필드 또는 `Blocker=N` 패턴을 파싱해 exit code로 반환.
- `run-epic.sh`의 최종 success path에서 호출. 실패 시 Epic 전체 exit code non-zero.

**P2-2. Reviewer review 로그 표준화 (E8)**
- 현재 slice review 로그가 "최종 한 줄"만 남는 경우가 있어 72분 같은 outlier의 시간대 분석 불가.
- Reviewer role에 "주요 판단 지점마다 timestamp 있는 줄을 남기라"(예: `[T+00:15] build pass`, `[T+00:45] diff 5/12 확인`) 정도의 경량 convention.

## 4. 검증 방법 (포지가 fix 후 확인)

각 개선 항목에 대해 다운스트림이 재현 가능한 테스트:
